# خارطة طريق إضافة Stitch (Roadmap)

## الأهداف

تُستخدم هذه الوثيقة لتسجيل اتجاهات التوسع المستقبلية لإضافة `stitch` خارج نطاق وثائق المواصفات الحالية، مما يساعد على التقدم على مراحل بدلاً من دفع جميع الإمكانيات في التنفيذ الحالي دفعة واحدة.

## المكتمل حالياً

- تفعيل/تعطيل الإضافة على مستوى المشروع
- حقن سير العمل بـ `stitch_design_review`
- بوابات التحقق `verify` والأرشفة `archive`
- أمر `ospec plugins run stitch`
- أوامر `ospec plugins approve stitch` / `reject stitch`
- أمر `ospec plugins doctor stitch`
- ملفات `approval.json` / `summary.md` / `result.json`
- الربط بين بروتوكول الذكاء الاصطناعي وتلميحات المهارات (skill prompts)
- مشغل افتراضي مدمج `Gemini CLI + stitch MCP`
- تلميحات تشخيصية واضحة عند فقدان Gemini CLI / stitch MCP / تلميح المصادقة

## المرحلة أ: محول Gemini CLI MCP

الهدف: تعميق مسار تكييف Gemini المدمج المتاح حالياً، وإكمال استقرار التنفيذ والقيود المعيارية الأقوى.

الحالة الحالية: تم تنفيذ أجزاء منها.

المحتوى المقترح:

- تثبيت عقود الإدخال والإخراج لـ `gemini-stitch-adapter`
- تحسين استقرار العقد الخاص بنتائج Gemini CLI العائدة إلى OSpec (`preview_url / summary_markdown / metadata / artifacts`)
- إضافة مسبارات أقوى قبل التنفيذ للتمييز بين "مثبت" و "قابل للاستدعاء حقاً"
- تقديم تلميحات أكثر دقة عندما لا يكون Gemini CLI مثبتاً محلياً، أو لم يتم تكوين `stitch` MCP، أو فقدان المصادقة، أو تعذر الوصول إلى Gemini

مطلوب تعاون المستخدم:

- التحقق في بيئة حقيقية من قدرة Gemini CLI على استدعاء stitch MCP بنجاح وإرجاع رابط معاينة قابل للمراجعة
- توفير كلمات تلميح موصى بها أو قيود استدعاء لـ stitch MCP إذا كانت هناك حاجة إلى عوائد هيكلية أكثر استقراراً

## المرحلة ب: محول Stitch API مدمج

الهدف: عدم الاعتماد على أغلفة خارجية، وتنفيذ طبقة تكييف Stitch API مدمجة مباشرة في OSpec.

المحتوى المقترح:

- قراءة تكوينات الرموز (tokens) على مستوى المشروع أو المستخدم مباشرة
- بدء طلبات إنشاء/تحديث صفحة Stitch مباشرة
- الاستعلام عن حالة المهمة وإنشاء رابط المعاينة النهائي `preview_url`
- كتابة أخطاء هيكلية في `result.json` عند الفشل

مطلوب تعاون المستخدم:

- توفير وثائق API مستقرة
- توفير طرق المصادقة وسياسات تحديد المعدل
- توفير دلالات أكواد الخطأ/حالة المهمة

## المرحلة ج: نواتج مراجعة أغنى

الهدف: جعل `stitch_design_review` أكثر من مجرد رابط، بل مجموعة من نواتج المراجعة.

المحتوى المقترح:

- لقطات شاشة للمعاينة
- رقم إصدار الصفحة أو معرف المراجعة (revision ID)
- ملخص التصميم الحالي
- فروقات التصميم (diff) / وصف التغييرات
- قوالب ملاحظات المراجعة اليدوية

## المرحلة د: إمكانيات متعددة (Multi Capability)

الهدف: دعم مهارات فرعية متعددة في إضافة `stitch` واحدة.

الإمكانيات المرشحة:

- `page_design_review`
- `component_design_review`
- `variant_comparison`
- `landing_page_generation`

كل إمكانية تحتاج إلى تعريف:

- اسم الخطوة
- أعلام التحفيز (flags)
- هل هي مانعة (blocking)
- هيكل الناتج
- موقع سير العمل الذي يجب أن يدرجه الذكاء الاصطناعي

## المرحلة هـ: تعاون الإضافات المتعددة

الهدف: ليس فقط دعم `stitch` بل دعم المزيد من الإضافات للتعاون في سير العمل.

المحتوى المقترح:

- تفعيل إضافات متعددة بالتوازي
- التحكم في ترتيب خطوات الإضافات
- حل النزاعات بين بوابات المنع المتشابهة
- تجريد مواصفات `result.json / approval.json` بشكل أكثر عمومية

## المرحلة و: فحص الصحة والقابلية للملاحظة

الهدف: جعل نظام الإضافات أسهل في استكشاف الأخطاء وإصلاحها والتشغيل.

المحتوى المقترح:

- فحص إمكانية الوصول لرابط المعاينة
- إحصائيات وقت تنفيذ المشغل (runner)
- ملخص لآخر تشغيل
- مخرجات أقوى لأمر `doctor`
- سجلات تصحيح أخطاء اختيارية

## الخطوة التالية الأكثر توصية حالياً

إذا كانت الأولوية لفعالية التنفيذ الحقيقي الحالية، فإن الترتيب المقترح هو:

1. تنفيذ "محول Gemini CLI MCP" أولاً
2. ثم إكمال "نواتج مراجعة أغنى"
3. وأخيراً النظر في "محول Stitch API مدمج"

السبب:

- جهازك الحالي مثبت عليه بالفعل Gemini CLI وتم تكوين stitch MCP
- هذا المسار هو الأقرب لطريقة استخدامك الحقيقية الآن
- يمكنه تشغيل دورة الإنتاج الكاملة "تغيير جديد ← معاينة Stitch ← قبول المستخدم ← متابعة التغيير" بأسرع وقت
