# تعليمات ترقية إصدار OSpec

تُستخدم هذه الوثيقة لتسجيل عملية ترقية الإصدار لمستودع الإصدارات هذا. في المستقبل، إذا قيل فقط "رفع الإصدار" أو "الرفع بإصدار واحد"، فسيتم التعامل مع ذلك كـ **patch +1** بشكل افتراضي، ما لم يتم تحديد رفع `minor` أو `major` بشكل صريح.

## نطاق التطبيق

- مستودع إصدارات OSpec CLI
- الحالات التي تتطلب مزامنة رقم إصدار الإصدار مع رقم الإصدار المعروض للجمهور
- يقوم المستخدم بتقديم (commit) Git بنفسه؛ هذه العملية لا تقوم بالتقديم تلقائياً

## القواعد الافتراضية

- `رفع إصدار واحد` = `patch +1`
- مثال: `0.6.0 -> 0.6.1`
- إذا ذكر المستخدم صراحةً رفع الإصدار الفرعي، فسيتم التعامل معه كـ `minor`
- إذا ذكر المستخدم صراحةً رفع الإصدار الرئيسي، فسيتم التعامل معه كـ `major`

## الملفات التي يجب مزامنتها

في كل مرة يتم فيها رفع الإصدار، يجب فحص ومزامنة المواقع التالية:

1. `package.json`
2. `package-lock.json`
3. `dist/cli.js`
4. `docs/usage.md`
5. `docs/usage.zh-CN.md`

## خطوات التشغيل القياسية

### 1. تأكيد الإصدار الحالي

تحقق أولاً من إصدار CLI الحالي في الملفات التالية:

- `package.json`
- `package-lock.json`
- `dist/cli.js`
- `docs/usage.md`
- `docs/usage.zh-CN.md`

### 2. ترقية `package.json` و `package-lock.json`

الأمر الموصى به:

```bash
npm version patch --no-git-tag-version
```

لرفع الإصدار الفرعي أو الرئيسي، استخدم على التوالي:

```bash
npm version minor --no-git-tag-version
```

```bash
npm version major --no-git-tag-version
```

### 3. مزامنة نسخة ثابت CLI (CLI Constant Version)

قم بتغيير `CLI_VERSION` في `dist/cli.js` إلى الإصدار الجديد.

### 4. مزامنة عرض الإصدار في الوثائق الخارجية

قم بتحديث عرض إصدار CLI في الملفات التالية:

- `docs/usage.md`
- `docs/usage.zh-CN.md`

لا يعرض README الرئيسي حالياً رقم إصدار CLI، لذا لا يُطلب تعديل README هنا.

يتم تحديث إصدار CLI فقط، ولا يتم تغيير إصدارات البروتوكولات أو المهارات (skills) الأخرى إلا إذا طلب ذلك بشكل منفصل.

### 5. إجراء التحقق الأدنى

نفذ على الأقل:

```bash
node dist/cli.js --version
```

تأكد من أن الإصدار الجديد الناتج صحيح.

عند الحاجة، يمكن تنفيذ ما يلي بالإضافة إلى ذلك:

```bash
npm run release:smoke
```

### 6. تعليمات التسليم

أبلغ المستخدم عند الانتهاء:

- من أي إصدار إلى أي إصدار تمت الترقية
- ما هي الملفات التي تم تعديلها
- هل تم إجراء التحقق الأدنى
- تقديم (commit) Git يتم بواسطة المستخدم نفسه

## اتفاقيات المستودع

- في المستقبل، إذا قال المستخدم فقط "رفع الإصدار" أو "الرفع بإصدار واحد"، فسيكون الافتراضي هو `patch +1`
- ما لم يطلب المستخدم صراحةً، لا يتم إنشاء علامة (tag) Git تلقائياً، ولا يتم التقديم (commit) تلقائياً لـ Git
- إذا تمت إضافة مواقع جديدة لعرض الإصدار في المستودع مستقبلاً، يجب إضافتها إلى هذه الوثيقة أيضاً
