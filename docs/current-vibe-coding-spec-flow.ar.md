# وثيقة عمل سير عمل Vibe Coding / Spec للمشروع الحالي

## 1. غرض الوثيقة

تُستخدم هذه الوثيقة لتنظيم عمليات المواصفات (spec) التي لا تزال مستخدمة بالفعل في المستودع الحالي، والتي يُعتزم الاحتفاظ بها.

التغييرات الجوهرية بعد هذه الجولة من التعديلات هي:

- لم يعد التهيئة ينقسم إلى خطوتين إلزاميتين: "تهيئة غلاف البروتوكول + أول توليد للوثائق docs generate".
- أمر `ospec init` الآن يأخذ المستودع مباشرة إلى حالة قابلة لتقديم التغييرات (change-ready).
- تم إنزال أمر `ospec docs generate` ليصبح أمراً للمتابعة والصيانة اللاحقة.
- تمت إزالة لوحة التحكم (dashboard)، وتوحيد سير العمل ليعود إلى واجهة سطر الأوامر (CLI).

## 2. المسار الرئيسي الحالي لسير العمل

يتقلص المشروع الحالي إلى مسار رئيسي مكون من أربع مراحل:

1. التهيئة إلى حالة "جاهز للتغيير" (change-ready)
2. تنفيذ التغيير النشط (active change)
3. النشر والتحقق
4. أرشفة المتطلبات

الأوامر الشائعة المقابلة هي:

```bash
ospec init [path]
ospec new <change-name> [path]
ospec progress [changes/active/<change>]
ospec verify [changes/active/<change>]
ospec finalize [changes/active/<change>]
```

إذا كنت تحتاج فقط لصيانة طبقة معرفة المشروع لاحقاً، استخدم:

```bash
ospec docs generate [path]
```

## 3. الحكم على الهيكل بعد التهيئة

في التنفيذ الحالي، لم يعد الحكم بعد التهيئة يعتمد على تسميات مستويات الهيكل، بل ينظر مباشرة إلى:

- هل تم التهيئة؟
- هل هو "جاهز للتغيير" (change-ready)؟
- هل تغطية الوثائق (docs) كاملة؟
- هل توجد تغييرات نشطة (active changes)؟
- هل التغيير الحالي قابل للأرشفة؟

## 4. طبقات المواصفات (Spec) الحالية

### 4.1 غلاف البروتوكول

يركز غلاف البروتوكول على قواعد التعاون نفسها، وتشمل الملفات الأساسية:

- `.skillrc`
- `.ospec/`
- `changes/active/`
- `changes/archived/`
- `SKILL.md`
- `SKILL.index.json`
- `for-ai/*`

### 4.2 طبقة معرفة المشروع

توضع طبقة معرفة المشروع طويلة الأمد بشكل أساسي في:

- `docs/project/overview.md`
- `docs/project/tech-stack.md`
- `docs/project/architecture.md`
- `docs/project/module-map.md`
- `docs/project/api-overview.md`

يتم إنشاء هذه الوثائق افتراضياً بواسطة `ospec init` لأول مرة؛ وإذا كانت هناك حاجة لتحديثها أو إصلاحها لاحقاً، يتم توكيل ذلك لأمر `ospec docs generate`.

### 4.3 مواصفات تنفيذ التغيير الواحد

ملفات البروتوكول الثابتة لكل تغيير نشط هي:

- `proposal.md`
- `tasks.md`
- `state.json`
- `verification.md`
- `review.md`

يبقى المصدر الحقيقي لحالة التنفيذ هو:

- `state.json`

## 5. ترتيب التنفيذ الحالي

الترتيب الموصى به للتغيير الواحد لا يزال كما يلي:

1. قراءة السياق والقيود
2. إنشاء أو تحديث `proposal.md`
3. إنشاء أو تحديث `tasks.md`
4. تنفيذ الكود
5. تحديث ملفات `SKILL.md` المتأثرة
6. إعادة بناء `SKILL.index.json`
7. إكمال `verification.md`
8. الأرشفة بعد تجاوز بوابة الأرشفة

## 6. العلاقة بين verify و archive

### `ospec verify`

يعمل حالياً كفحص مسبق، يتحقق بشكل أساسي من:

- وجود `proposal.md / tasks.md / verification.md`
- تغطية الخطوات الاختيارية المفعلة بالوثائق
- ما إذا كانت قائمة التحقق لا تزال تحتوي على بنود غير محددة
- ما إذا كانت الحالة الحالية في `state.json` منطقية

### `ospec archive`

هو بوابة الأرشفة الحقيقية حالياً، وتكون متطلباته أكثر صرامة:

- `state.json.status == ready_to_archive`
- تم تجاوز التحقق (`verification_passed`)
- تم تحديث المهارات (`skill_updated`)
- تم إعادة توليد الفهرس (`index_regenerated`)
- دخلت الخطوات الاختيارية المفعلة في `passed_optional_steps`
- لم يعد هناك بنود غير محددة في `tasks.md` و `verification.md`

### `ospec finalize`

لا يزال هو المدخل الرسمي القياسي للإغلاق، ويُستخدم لربط:

- التحقق (verify)
- تحديث الفهرس (index refresh)
- الأرشفة (archive)

## 7. التهيئة والتكملة والمتابعة عبر الذكاء الاصطناعي

عندما يعبر المستخدم عبر تلميح الذكاء الاصطناعي عن "تهيئة المشروع باستخدام OSpec"، فإن السلوك الموصى به حالياً هو:

1. الدخول مباشرة في سير عمل التهيئة
2. إذا كانت وثائق وصف المشروع موجودة بالفعل، يتم إعادة استخدامها مباشرة
3. إذا كان السياق الكافي مفقوداً، يتم السؤال مرة واحدة فقط عن ملخص المشروع أو التقنيات المستخدمة
4. إذا لم يقم المستخدم بالإكمال، يتم الاستمرار في التهيئة على أي حال، وتوليد وثائق مؤقتة بانتظار التكملة
5. بعد اكتمال التهيئة، يجب أن يكون المستودع جاهزاً مباشرة لأمر `ospec new`

عندما ينفذ المستخدم `ospec init` مباشرة في الطرفية (terminal):

- لا يتم طرح أسئلة متابعة حوارية
- يتم وضع وثائق المشروع المؤقتة مباشرة
- ضمان أن النتيجة لا تزال "جاهزة للتغيير" (change-ready)

## 8. العناصر المحذوفة في هذه الجولة

### 8.1 لوحة التحكم (Dashboard)

تمت إزالة الأكواد والأوامر المتعلقة بلوحة التحكم.

المستودع الحالي لا يحتفظ بـ:

- مدخل أمر dashboard
- كود خادم dashboard
- أصول الواجهة الأمامية الثابتة لـ dashboard
- نصوص المساعدة المتعلقة بـ dashboard

### 8.2 تسميات مستويات الهيكل

عند مناقشة سير العمل لاحقاً، لن يتم استخدام تسميات مستويات الهيكل.

يتم التوحيد لمناقشة:

- هل تم التهيئة؟
- هل هو "جاهز للتغيير" (change-ready)؟
- هل طبقة المعرفة كاملة؟
- هل التغيير قيد التنفيذ؟
- هل هو قابل للأرشفة بالفعل؟
