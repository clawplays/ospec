# نظرة عامة على المشروع

## ما هو OSpec

OSpec هو سير عمل مدفوع بالوثائق للتطوير المدعوم بالذكاء الاصطناعي، مما يساعدك على تحديد المتطلبات والتغييرات في الوثائق أولاً، ثم دفع التنفيذ والتحقق والأرشفة من خلال التعاون مع الذكاء الاصطناعي.

يعامل الوثائق كمصدر وحيد للحقيقة للمتطلبات والتغييرات. تقوم التهيئة بإعداد قواعد التعاون ومعرفة المشروع الأساسية أولاً، ثم تسمح للذكاء الاصطناعي بدفع التنفيذ من خلال حاويات تغيير صريحة.

## المبادئ الأساسية

- يجب أن يترك `ospec init` المستودع في حالة جاهزة للتغيير (change-ready).
- يمكن للتهيئة المدعومة بالذكاء الاصطناعي طلب متابعة موجزة واحدة لملخص المشروع أو مكدس التقنيات عند فقدان سياق المستودع.
- تبقى التهيئة البسيطة عبر واجهة سطر الأوامر غير تفاعلية وترجع إلى وثائق المشروع المؤقتة عند عدم توفر السياق.
- `ospec docs generate` هو أمر صيانة لاحق لتحديث أو إصلاح أو ملء وثائق معرفة المشروع.
- التغيير الأول لا يزال صريحاً. التهيئة لا تنشئ عمل تنفيذ تلقائياً.
- المسار الموصى به للمستخدم هو `init -> execute -> deploy/validate -> archive`.

## حالات الاستخدام النموذجية

- المستودعات التي تريد سلوكاً متسقاً للذكاء الاصطناعي
- الفرق التي تريد أن يكون تنفيذ التغيير مرئياً وقابلاً للمراجعة
- المشاريع التي قد تصبح مستودعات ويب أو واجهة سطر أوامر أو Unity أو Godot أو خلفية أو بروتوكول فقط
- البيئات التي تحتاج إلى نفس سير العمل عبر Codex و Claude Code

## القدرات الحالية

- تهيئة جاهزة للتغيير
- صيانة معرفة المشروع
- إدارة سير عمل التغيير النشط
- مسار إغلاق قياسي `finalize -> archive -> commit-ready`
- حالة تغيير مجمعة `PASS / WARN / FAIL`
- فرض خطافات Git للتغييرات النشطة
- تثبيت ومزامنة مهارات Codex و Claude Code

## نموذج الأوامر

لدليل جديد، الترتيب المعتاد هو:

```bash
ospec init [path]
ospec new <change-name> [path]
ospec verify [changes/active/<change>]
ospec finalize [changes/active/<change>]
```

عندما تحتاج فقط إلى صيانة الوثائق لاحقاً، استخدم:

```bash
ospec docs generate [path]
```
