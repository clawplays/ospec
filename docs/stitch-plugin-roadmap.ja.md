# Stitch プラグイン ロードマップ

## 目標

本文書は、現在の仕様文書以外の `stitch` プラグインの今後の拡張方向を記録し、すべての機能を一度に実装するのではなく、段階的に推進するのに役立てるためのものです。

## 現在完了していること

- プロジェクトレベルのプラグイン有効化/無効化
- ワークフローへの `stitch_design_review` の注入
- `verify` / `archive` のゲート機能
- `ospec plugins run stitch` コマンド
- `ospec plugins approve stitch` / `reject stitch` コマンド
- `ospec plugins doctor stitch` コマンド
- `approval.json` / `summary.md` / `result.json` の管理
- AIプロトコルとスキルのプロンプト連携
- 内蔵の `Gemini CLI + stitch MCP` デフォルトランナー
- Gemini CLI / stitch MCP / 認証ヒントが不足している場合の明確な診断プロンプト

## フェーズ A：Gemini CLI MCP アダプター

目標：現在利用可能な内蔵の Gemini アダプターパスをさらに深化させ、実行の安定性と標準化された制約を補完します。

現在のステータス：一部実装済み。

推奨内容：

- `gemini-stitch-adapter` の入出力コントラクトの固化
- Gemini CLI の返却結果から OSpec の `preview_url / summary_markdown / metadata / artifacts` コントラクトへの安定性のさらなる向上
- 「インストール済み」と「実際に呼び出し可能」を区別する、より強力な実行前プローブの追加
- ローカルに Gemini CLI がインストールされていない、`stitch` MCP が設定されていない、認証が不足している、または Gemini アップストリームに到達できない場合に、より正確なヒントを表示

ユーザーの協力が必要な点：

- 実際の環境で Gemini CLI が stitch MCP を正常に呼び出し、レビュー可能なプレビュー URL を返せることを検証する
- より安定した構造化データが必要な場合、stitch MCP の推奨プロンプトや呼び出し制約を提供する

## フェーズ B：内蔵 Stitch API アダプター

目標：外部のラッパーに依存せず、OSpec 内に直接 Stitch API アダプターレイヤーを組み込みます。

推奨内容：

- プロジェクトまたはユーザーレベルのトークン設定を直接読み込む
- Stitch ページの生成/更新リクエストを直接発行する
- タスクステータスをポーリングし、最終的な `preview_url` を生成する
- 失敗時に `result.json` に構造化されたエラーを書き込む

ユーザーの協力が必要な点：

- 安定した API ドキュメントの提供
- 認証方法とレート制限ポリシーの提供
- エラーコード/タスクステータスのセマンティクスの提供

## フェーズ C：より豊富なレビュー成果物

目標：`stitch_design_review` を単なる URL ではなく、一連のレビュー可能な成果物に昇華させます。

推奨内容：

- プレビューのスクリーンショット
- ページバージョン番号またはリビジョン ID
- 今回のデザイン概要
- デザインの差分（diff）/ 変更説明
- 手動レビュー用のメモテンプレート

## フェーズ D：マルチ Capability

目標：1つの `stitch` プラグインで複数のサブ機能をサポートします。

候補となる Capability：

- `page_design_review`
- `component_design_review`
- `variant_comparison`
- `landing_page_generation`

各 Capability で定義が必要な事項：

- ステップ名
- トリガーフラグ
- ブロッキング（遮断）するかどうか
- 成果物の構造
- AI が挿入すべきフローの位置

## フェーズ E：マルチプラグインの協調

目標：`stitch` だけでなく、複数のプラグインがワークフロー内で共存・協調できるようにします。

推奨内容：

- 複数のプラグインの並列有効化
- プラグインステップの順序制御
- 同種のブロッキングゲートの衝突解決
- より汎用的な `result.json / approval.json` 仕様の抽象化

## フェーズ F：ヘルスチェックと観測可能性

目標：プラグインシステムのトラブルシューティングと運用を容易にします。

推奨内容：

- プレビュー URL の到達性チェック
- ランナーの実行時間統計
- 直近の実行概要
- `doctor` 出力の強化
- オプションのデバッグログ

## 現在最も推奨される次のステップ

現在の実際の導入効率を優先する場合、推奨される順序は以下の通りです：

1. まず「Gemini CLI MCP アダプター」を完成させる
2. 次に「より豊富なレビュー成果物」を補完する
3. 最後に「内蔵 Stitch API アダプター」を検討する

理由：

- 現在のPCにはすでに Gemini CLI がインストールされ、stitch MCP が設定されている
- このパスが現在の実際の使用方法に最も近い
- 「新しい変更 → Stitch プレビュー → ユーザー確認 → 変更の継続」という完全なプロダクションフローを最速で構築できる
